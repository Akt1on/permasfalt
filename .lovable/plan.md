## Goal

Convert the project from TanStack Start (Cloudflare Worker SSR) to a plain Vite SPA deployable on Vercel, with backend logic moved to `api/*.ts` serverless functions.

## Key decisions

1. **Router**: Keep `@tanstack/react-router` with file-based routing (works fine in SPA mode via `@tanstack/router-plugin/vite`). All existing `createFileRoute(...)` files keep working; only the root shell changes. Avoids rewriting every route.
2. **Head/SEO tags**: Replace `HeadContent` + per-route `head()` with `react-helmet-async`. Each route gets a `<Helmet>` block instead of a `head()` config. SEO meta still works client-side (Vercel serves static HTML; crawlers like Google execute JS).
3. **Server functions → Vercel API**: Each `createServerFn` becomes a POST `api/*.ts` handler using `@vercel/node` types. Client calls via `fetch('/api/...')`. Supabase admin client moves to `api/_lib/supabase-admin.ts` (server-only).
4. **Auth on API routes**: Client sends `Authorization: Bearer <access_token>` header (read from `supabase.auth.getSession()`); API routes validate via `supabase.auth.getClaims(token)`.
5. **No SSR-only files**: Delete `src/server.ts`, `src/start.ts`, `src/lib/error-capture.ts`, `src/lib/error-page.ts`, `wrangler.jsonc`, `auth-middleware.ts`, `client.server.ts`, `auth-attacher.ts`.

## File changes

### Delete
- `src/server.ts`, `src/start.ts`
- `src/lib/error-capture.ts`, `src/lib/error-page.ts`
- `src/integrations/supabase/{client.server,auth-middleware,auth-attacher}.ts`
- `wrangler.jsonc`
- `src/lib/leads.functions.ts`, `src/lib/admin.functions.ts` (replaced by `api/*`)

### Create
- `index.html` (root) — Vite SPA entry
- `src/main.tsx` — bootstraps `RouterProvider` + `HelmetProvider` + `QueryClientProvider`
- `vercel.json` — SPA rewrite (`/((?!api/).*) → /index.html`) and Node runtime for `/api/*`
- `api/_lib/supabase-admin.ts` — service-role client (env: `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`)
- `api/_lib/auth.ts` — `requireAdmin(req)` and `getUserFromRequest(req)` helpers
- `api/leads.ts` — POST: validate + insert lead + send Telegram
- `api/admin/me.ts` — GET: admin status (bootstrap first user)
- `api/admin/leads.ts` — GET: list leads; PATCH: update status

### Rewrite
- `vite.config.ts` — use plain `defineConfig` from `vite` + `@vitejs/plugin-react` + `@tanstack/router-plugin/vite` + `tailwindcss` + path alias. No Lovable tanstack-start config, no Cloudflare plugin.
- `src/routes/__root.tsx` — remove `shellComponent`, `HeadContent`, `Scripts`; just `<HelmetProvider>` + `<Outlet/>`. HTML shell moves to `index.html`.
- `src/router.tsx` — keep as-is (already SPA-compatible).
- Every route file with `head: () => ({...})` — replace with `<Helmet>` inside the component (8 routes: index, ceny, kontakty, o-nas, obekty, otzyvy, politika-konfidencialnosti, uslugi).
- `src/components/ContactForm.tsx` — replace `useServerFn(submitLead)` with `fetch('/api/leads', { method: 'POST', body: JSON.stringify(...) })`.
- `src/routes/admin.tsx`, `admin.index.tsx`, `admin.login.tsx` — replace `useServerFn` calls with `fetch('/api/admin/*', { headers: { Authorization: \`Bearer ${token}\` } })`.

### Package changes
- **Remove**: `@tanstack/react-start`, `@cloudflare/vite-plugin`, `wrangler`, `@lovable.dev/vite-tanstack-config`
- **Add**: `vite`, `@vitejs/plugin-react`, `@tanstack/router-plugin`, `react-helmet-async`, `@vercel/node`, `@tailwindcss/vite`

## vercel.json

```json
{
  "rewrites": [{ "source": "/((?!api/).*)", "destination": "/index.html" }],
  "functions": { "api/**/*.ts": { "runtime": "@vercel/node@3" } }
}
```

## Env vars for Vercel

User must set in Vercel dashboard:
- `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`
- `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` (client build-time)
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` (optional)

## Caveats the user should know

1. **Lovable Cloud auto-managed files are gone.** `src/integrations/supabase/client.ts` is auto-generated by Lovable and will be regenerated on next sync — but `client.server.ts` / `auth-middleware.ts` / `auth-attacher.ts` may also be regenerated. After this refactor, the Lovable editor's preview will likely break because the Lovable runtime expects TanStack Start. **The project will only work when deployed to Vercel, not in the Lovable preview.** Confirm you're OK with that.
2. **SEO**: Without SSR, initial HTML has no per-page meta tags. Google's crawler runs JS so it's fine for indexing, but social-media link previews (FB/Twitter cards) will see only the default `index.html` meta. If that matters, we'd need to keep SSR or add a build-time pre-render step (not in this plan).
3. **Lovable Cloud integration still works** for the database — only the server-side code execution moves to Vercel functions.
